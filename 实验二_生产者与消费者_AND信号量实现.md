## 📚 理论基础

### 1. 生产者-消费者问题

生产者-消费者问题是操作系统中经典的进程同步问题：
- **生产者进程**：生产产品并放入缓冲区
- **消费者进程**：从缓冲区取出产品并消费
- **缓冲区**：有限容量的共享资源，需要互斥访问

**核心约束**：
1. 缓冲区满时，生产者必须等待
2. 缓冲区空时，消费者必须等待
3. 缓冲区的访问必须互斥（同一时间只能有一个进程访问）

### 2. AND 信号量机制

AND 信号量是一种高级同步机制，允许进程**原子性地申请或释放多个信号量**。

#### Swait 操作（同时等待）
```
Swait(S1, S2, ..., Sn)
```
- 要么同时获取所有信号量，要么一个都不获取
- 避免了传统信号量可能导致的死锁问题
- 实现了资源的原子性分配

#### Ssignal 操作（同时发送）
```
Ssignal(S1, S2, ..., Sn)
```
- 同时释放所有信号量
- 唤醒等待这些信号量的进程

### 3. 本实验的信号量设计

使用三个信号量：

| 信号量 | 初始值 | 含义 |
|--------|--------|------|
| mutex  | 1      | 互斥信号量，保证对缓冲区的互斥访问 |
| empty  | n      | 资源信号量，表示空闲缓冲区数量（n=5） |
| full   | 0      | 资源信号量，表示已填充缓冲区数量 |

#### 生产者操作流程
```
while (true) {
    生产产品;
    Swait(empty, mutex);  // P操作：申请空位和互斥锁
    将产品放入缓冲区;      // 临界区
    Ssignal(mutex, full); // V操作：释放互斥锁并增加满位
}
```

#### 消费者操作流程
```
while (true) {
    Swait(full, mutex);   // P操作：申请满位和互斥锁
    从缓冲区取出产品;      // 临界区
    Ssignal(mutex, empty);// V操作：释放互斥锁并增加空位
    消费产品;
}
```

**关键点**：
- ✅ 先申请资源信号量（empty/full），再申请互斥信号量（mutex）
- ✅ 避免死锁：如果只拿到 mutex 但拿不到 empty，会释放 mutex 并重试
- ✅ 原子性：Swait 要么全部成功，要么全部失败

### 核心类说明

#### 1. AndSemaphore 类
**位置**：`src/main/java/com/homework/osexperiment/osexp/model/AndSemaphore.java`

**功能**：实现 AND 信号量的 Swait/Ssignal 操作

**核心方法**：
- `swait()`：原子性地获取所有信号量
- `ssignal()`：同时释放所有信号量
- `getSemaphoreValue(index)`：获取指定信号量的当前值

**实现原理**：
```java
public void swait() throws InterruptedException {
    lock.lock();
    try {
        while (!allAcquired) {
            // 尝试获取所有信号量
            for (Semaphore sem : semaphores) {
                if (sem.tryAcquire()) {
                    acquiredCount++;
                } else {
                    break;
                }
            }
            
            // 如果全部获取成功
            if (acquiredCount == semaphores.length) {
                allAcquired = true;
            } else {
                // 失败则释放已获取的信号量，等待后重试
                releaseAll(acquiredCount);
                Thread.sleep(50); // 短暂等待后重试
            }
        }
    } finally {
        lock.unlock();
    }
}
```

#### 2. Buffer 类
**位置**：`src/main/java/com/homework/osexperiment/osexp/model/Buffer.java`

**功能**：管理缓冲区并使用 AND 信号量控制访问

**核心方法**：
- `produce(Product)`：生产者调用，执行完整的 P/V 操作
- `consume()`：消费者调用，执行完整的 P/V 操作
- `getMutexValue()`、`getEmptyValue()`、`getFullValue()`：获取信号量当前值

**PV 操作日志**：
每次生产或消费都会输出详细的 PV 操作过程：
```

[P操作前] mutex=1, empty=5, full=0
[P操作] Swait(empty, mutex) - 申请空缓冲区和互斥锁
[AND信号量-Swait] 成功获取: empty, mutex, 
[P操作后] mutex=0, empty=4, full=0
[临界区] 生产了产品 #1，当前库存：1
[V操作] Ssignal(mutex, full) - 释放互斥锁并增加满缓冲区
[AND信号量-Ssignal] 释放: mutex, full, 
[V操作后] mutex=1, empty=4, full=1
========== 生产者完成生产 ==========
```

**核心特性**：

### 观察要点

#### 1. 信号量变化规律

**正常生产时**：
```
操作前: mutex=1, empty=5, full=0
Swait(empty, mutex) → mutex=0, empty=4, full=0
[生产产品]
Ssignal(mutex, full) → mutex=1, empty=4, full=1
```

**正常消费时**：
```
操作前: mutex=1, empty=4, full=1
Swait(full, mutex) → mutex=0, empty=4, full=0
[消费产品]
Ssignal(mutex, empty) → mutex=1, empty=5, full=0
```

#### 2. 边界情况观察

**缓冲区满时**：
- 生产者会在 `Swait(empty, mutex)` 处阻塞
- PV 操作日志显示："⚠️ 缓冲区已满，生产者在 Swait(empty) 处阻塞等待"
- empty 信号量值为 0

**缓冲区空时**：
- 消费者会在 `Swait(full, mutex)` 处阻塞
- PV 操作日志显示："⚠️ 缓冲区为空，消费者在 Swait(full) 处阻塞等待"
- full 信号量值为 0

**互斥访问**：
- 当 mutex=0 时，表示有进程在临界区
- 缓冲区边框变为红色（生产者）或蓝色（消费者）
- 另一个进程无法进入临界区

#### 3. 控制台日志分析

每次生产或消费都会输出完整的 PV 操作流程：

```
========== 生产者开始生产 ==========
[P操作前] mutex=1, empty=3, full=2
[P操作] Swait(empty, mutex) - 申请空缓冲区和互斥锁
[AND信号量-Swait] 成功获取: empty, mutex, 
[P操作后] mutex=0, empty=2, full=2
[临界区] 生产了产品 #4，当前库存：3
[V操作] Ssignal(mutex, full) - 释放互斥锁并增加满缓冲区
[AND信号量-Ssignal] 释放: mutex, full, 
[V操作后] mutex=1, empty=2, full=3
========== 生产者完成生产 ==========
```

**日志解读**：
1. **P操作前**：显示操作前的信号量状态
2. **P操作**：说明正在执行 Swait 操作
3. **Swait 成功**：确认原子性地获取了所有信号量
4. **P操作后**：显示 P 操作后的信号量状态
5. **临界区**：显示实际的生产/消费行为
6. **V操作**：说明正在执行 Ssignal 操作
7. **Ssignal 完成**：确认释放了所有信号量
8. **V操作后**：显示最终的信号量状态

## 🔍 实验验证

### 验证点 1：互斥性

**测试步骤**：
1. 启动生产者，观察 mutex 信号量
2. 在生产者进入临界区时（mutex=0），立即启动消费者
3. 观察消费者是否被阻塞

**预期结果**：
- 消费者会等待生产者离开临界区
- 控制台显示："资源不足，等待中..."
- 直到生产者执行 Ssignal 释放 mutex

### 验证点 2：同步性

**测试步骤**：
1. 连续生产 5 个产品，使缓冲区满（empty=0）
2. 尝试继续生产

**预期结果**：
- 生产者被阻塞在 Swait(empty, mutex)
- PV 操作日志显示："⚠️ 缓冲区已满，生产者在 Swait(empty) 处阻塞等待"
- 启动消费者后，生产者才能继续

### 验证点 3：AND 信号量的原子性

**测试步骤**：
1. 观察控制台日志中的 Swait 操作
2. 检查是否出现只获取部分信号量的情况

**预期结果**：
- 日志只显示"成功获取: empty, mutex"或"资源不足，等待中..."
- 不会出现只获取 empty 但未获取 mutex 的情况
- 体现了 AND 信号量的原子性特征

## 💡 实验总结

### 关键收获

1. **AND 信号量的优势**
   - 避免了传统信号量的死锁问题
   - 实现了资源的原子性分配
   - 简化了同步逻辑

2. **PV 操作的本质**
   - P 操作（wait）：申请资源，可能阻塞
   - V 操作（signal）：释放资源，唤醒等待进程
   - 成对出现，保证资源平衡

3. **进程同步的核心**
   - 互斥：通过 mutex 信号量实现
   - 同步：通过 empty/full 信号量实现
   - 阻塞与唤醒：自动由信号量机制处理

### 技术亮点

1. ✅ **真正的多线程并发**：生产者和消费者在独立线程中运行
2. ✅ **原子性保证**：Swait 操作确保要么全部获取，要么全部不获取
3. ✅ **实时可视化**：信号量值和 PV 操作实时显示
4. ✅ **详细日志**：控制台输出完整的 PV 操作流程
5. ✅ **线程安全**：所有 UI 更新通过 Platform.runLater() 确保安全

### 改进方向

1. 支持多个生产者和多个消费者同时运行
2. 添加性能统计（平均等待时间、吞吐量等）
3. 实现更复杂的调度策略
4. 添加死锁检测机制
